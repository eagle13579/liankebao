# ADR-009: 多租户生产级迁移 — SQLite → PostgreSQL + RLS

- **状态**: 已批准
- **日期**: 2026-06-10
- **决策者**: 向海容 · 白泽 · 烛龙 · 狴犴
- **技术负责人**: 文鳐
- **标签**: 多租户, 数据库, RLS, 生产迁移, 隔离

---

## 1. 上下文

链客宝当前生产环境（47.116.116.87）使用 **SQLite + 应用层隔离** 运行多租户逻辑。随着数字名片、B2B交易撮合、产品展示等核心场景的客户数增长，该架构暴露出以下瓶颈：

| 问题 | 现状 | 影响 |
|------|------|------|
| 写锁串行 | SQLite 单写者模型，WAL 模式缓解有限 | 并发请求 >5 TPS 即出现 `database is locked` |
| 无数据库级隔离 | `TenantSessionWrapper` 应用层过滤，纯靠代码约束 | 开发疏忽可导致跨租户数据泄漏 |
| 无行级权限 | 无法按行粒度控制访问 | 报表、API 导出等场景需额外过滤层 |
| 单机瓶颈 | SQLite 无法水平扩展 | 数据量 >10GB 后性能骤降 |
| 无 PITR | SQLite 备份为文件快照，无法时间点恢复 | 数据丢失风险高 |

**迁移目标**: 在保持现有业务模型不变的前提下，将数据库从 SQLite 切换为 PostgreSQL，并启用原生 Row-Level Security (RLS) 实现数据库层强制租户隔离。

---

## 2. 备选方案对比

### 方案 A: SQLite + 应用层隔离（当前方案 ❌ 不选）

| 维度 | 评估 |
|------|------|
| 并发能力 | 无。`PRAGMA busy_timeout=5000` 仅缓解，仍串行写 |
| RLS | 不支持，完全依赖 `TenantSessionWrapper` 应用过滤 |
| 性能 | 单文件，>10GB 后 B-tree 退化，查询 >50ms |
| 备份 | 文件快照，需停写或无原子性保证 |
| 生态 | 无连接池、无主从、无 PITR |
| 综合 | 适用于开发/单用户，不适合生产 SaaS 多租户 |

### 方案 B: MySQL + 应用层隔离 ❌ 不选

| 维度 | 评估 |
|------|------|
| 并发能力 | ✅ InnoDB 行锁，高并发支持 |
| RLS | ❌ MySQL 无行级安全策略（社区版），需应用层 + 视图层双重过滤 |
| JSON 支持 | ⚠️ MySQL 5.7+ 虽支持 JSON，但 JSON 索引、路径查询能力弱于 PG |
| 隔离实现 | 需自行实现 `WHERE organization_id = ?` 拦截层，与方案 A 相同风险 |
| 迁移成本 | 现有 `_org_fk()` 列兼容，但多租户逻辑无实质性改善 |
| 综合 | 并发能力有提升，但隔离机制无本质变化，不解决核心问题 |

### 方案 C: PostgreSQL + RLS（选定 ✅）

| 维度 | 评估 |
|------|------|
| 并发能力 | ✅ MVCC + 连接池，并发 >1000 TPS |
| RLS | ✅ 原生 `CREATE POLICY` 按行级控制，数据库层强制隔离 |
| JSON 支持 | ✅ JSONB + GIN 索引，适合非结构化产品规格/标签查询 |
| 隔离实现 | 无需应用层干预，`current_setting('app.current_tenant_id')` 自动过滤 |
| 生态路线 | 可扩展为 RDS → Aurora → Citus 分布式 |
| 综合 | 最适合多租户 SaaS 的数据库方案 |

---

## 3. 决策理由

**RLS 原生支持是核心决策因素**，具体理由如下：

1. **数据库层强制隔离** — 应用层过滤（`TenantSessionWrapper`）可被绕过（如原生 SQL 查询、批量导出脚本、ORM 逃生口）。RLS 在查询计划阶段附加行级谓词，**无论通过何种接口访问，均强制隔离**。

2. **链客宝的租户场景要求严格隔离** — 数字名片数据含用户隐私联系方式、B2B 交易含价格/佣金敏感信息、产品展示涉及商业机密。任一跨租户泄漏即可造成法律风险与信任危机。

3. **开发安全冗余** — RLS 作为安全兜底层：即便开发者在新增路由时忘记调用 `apply_tenant_filter()`，RLS 策略仍会拦截。这与"纵深防御"原则一致。

4. **JSONB 原生支持** — 产品 `specs`、`details`、`tags` 等字段已是 JSON 格式，当前用 `json.dumps()` 存为字符串。PG JSONB 允许直接索引与查询，未来可支持按标签/category 的租户内全文检索。

5. **成熟生态与扩展路线** — 从单机 Docker PG → 阿里云 RDS PostgreSQL → PG 主从 + 读写分离 → Citus 分布式，平缓扩展无需换数据库。

6. **社区验证** — GitLab、Discourse、Supabase 等知名多租户 SaaS 均采用 PG + RLS 方案。

## 4. 二阶效应

### 4.1 数据迁移 — 存量用户适配

- **影响范围**: 现有 SQLite 中的 18 张业务表均通过 `_org_fk()` 定义了 `organization_id` 列，但存量数据该列为 `NULL`（SQLite 模式下 `_IS_MULTI_TENANT` 为 False 时 `_org_fk()` 返回 `Column(Integer, nullable=True, default=None)`）。
- **迁移任务**:
  1. 将 SQLite 数据使用 `pgloader` 或自定义 ETL 迁移至 PG
  2. 为每行数据填充 `organization_id`：按 `owner_id` → `user_id` → 默认组织映射，确保无 NULL
  3. 将 `_org_fk()` 定义中 `nullable=True` 改为 `nullable=False`（PG 模式下）
- **风险**: 若某个存量行无法映射到组织，迁移将失败。需准备手工映射表。

### 4.2 部署变更

| 变更项 | 说明 |
|--------|------|
| 新增 Docker Compose 服务 | `postgres:15` 容器，初始化 RLS 函数与策略 |
| CI/CD 流水线 | 新增 `run_migrations.sh` 步骤 — 创建 RLS 策略 |
| 环境变量 | 所有环境新增 `DB_TYPE=postgres` 及 `DATABASE_URL` |
| 存储卷 | 生产服务器新增 PG 数据卷 `/var/lib/postgresql/data` |

### 4.3 回滚方案

若迁移后 48h 内发现严重问题，回滚步骤：

```
1. 环境变量: 改回 DB_TYPE=sqlite（不设置）
2. 代码回退: 回滚 RLS 相关代码变更
3. 数据: 保留 SQLite 源文件（迁移期间不做删除）
4. 验证: 走通核心业务（登录 → 浏览产品 → 下单）
```

**注**: 回滚后 RLS 策略定义的 `CREATE POLICY` SQL 留在 PG 中但不会被执行；应用层重新启用 `TenantSessionWrapper` 过滤。

## 5. 实施顺序

### Step 1a — 部署 PostgreSQL （1~2 天）

- [ ] 生产服务器安装 Docker，拉取 `postgres:15-alpine`
- [ ] 配置 `docker-compose.yml`，挂载数据卷、设置密码、端口映射
- [ ] 创建数据库 `chainke_prod` 和专用用户 `chainke_app`
- [ ] 配置 `pg_hba.conf` 仅允许应用服务器 IP 连接
- [ ] 部署 RLS 辅助函数 `set_tenant_context()` 和策略模板
- [ ] 验证连接（使用 `psql` 或应用 `init_db()`）

### Step 1b — 数据迁移 + 填充 org_id （2~3 天）

- [ ] 停止应用写操作（或切维护模式）
- [ ] 使用 ETL 脚本将 SQLite 数据批量迁移至 PG
- [ ] 为每条业务记录填充 `organization_id`（按用户 → 组织映射规则）
- [ ] 将 `_org_fk()` 在 PG 模式下改为 `nullable=False`
- [ ] 添加 `ForeignKey` 约束验证
- [ ] 运行数据一致性校验（行数对比、外键完整性检查）

### Step 1c — 打开多租户开关 + 启用 RLS （1 天）

- [ ] 设置环境变量 `DB_TYPE=postgres` 和 `IS_MULTI_TENANT=true`
- [ ] 应用重启，确认 `is_multi_tenant()` 返回 true
- [ ] 启用 RLS: `ALTER TABLE ... ENABLE ROW LEVEL SECURITY`
- [ ] 创建 RLS 策略: `CREATE POLICY tenant_isolation ON ... USING (organization_id = current_setting('app.current_tenant_id')::int)`
- [ ] 验证跨租户隔离: 不同 org 用户查询不应互见数据
- [ ] 灰度放量: 选 1 个 test org 全流程验证 → 再全量切换

## 6. RLS 策略设计（概要）

```sql
-- 为每张业务表启用 RLS
ALTER TABLE users ENABLE ROW LEVEL SECURITY;
ALTER TABLE products ENABLE ROW LEVEL SECURITY;
ALTER TABLE orders ENABLE ROW LEVEL SECURITY;
-- ... 其余 15 张表相同

-- 租户隔离策略（SELECT / INSERT / UPDATE / DELETE）
CREATE POLICY tenant_isolation ON products FOR ALL
    USING (organization_id = current_setting('app.current_tenant_id')::int)
    WITH CHECK (organization_id = current_setting('app.current_tenant_id')::int);

-- 管理员豁免策略（超级租户可跨组织查询）
CREATE POLICY admin_bypass ON products FOR ALL
    USING (current_setting('app.is_admin')::boolean IS TRUE);
```

**应用侧配合**: FastAPI 中间件在请求入口调用 `SELECT set_config('app.current_tenant_id', :org_id, true)`。
