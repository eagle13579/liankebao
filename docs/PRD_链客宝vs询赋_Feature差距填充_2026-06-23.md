## 1. 背景与调研摘要

### 1.1 竞品概述

询赋是一款iOS原生App（App Store ID 6737821838），定位为「人脉资源管理商机匹配AI助手」。其核心理念是「身份多元—资源重组—价值共创」，核心差异化在：多身份角色系统 + 社交图谱信任匹配 + 资源变现飞轮。

### 1.2 链客宝当前定位

链客宝是**B2B企业家供需匹配双边网络平台**，核心模式为「产品方上架→推广员分销→买家下单→分润结算」。当前MVP已打通交易全链路（后端API 100%覆盖），匹配引擎采用 **B3方案（预计算embedding查表）+ 三塔DNN + Thompson sampling Bandit** 架构。极端重视数据安全（韩国真实企业数据不离开内网）。

### 1.3 提取的8个可吸收Feature

| 编号 | Feature | 原始优先级 | B2B化改造命名 |
|:----:|---------|:----------:|:-------------:|
| F1 | 多身份角色系统 | P1 | 企业多角色身份系统 |
| F2 | AI个人资源数据库 | P2 | 企业能力自动标签库 |
| F3 | 熟人资源圈（社交图谱匹配） | P1 | 社交信任链权重匹配 |
| F4 | 一键多平台分发 | P2 | 一键多平台分发 |
| F5 | 资源/需求发布市场 | P0 | 需求发布市场 |
| F6 | 私单/合作记录沉淀 | P2 | B2B合作案例沉淀 |
| F7 | MCP标准化匹配协议 | P2 | 匹配协议标准化 |
| F8 | 批量导入建联 | P1 | 批量导入企业信息 |

### 2.1 差距等级说明

- **🔴 缺失**: 链客宝完全没有此能力，需从0开始建设
- **🟡 部分**: 链客宝有近似能力，但差距明显，需升级改造
- **🟢 已有**: 已覆盖或适配后即可使用

### 2.2 优先级判定逻辑

- **P0 (核心差异化)**: 直接影响核心价值主张和用户体验，缺失将导致产品竞争力严重不足
- **P1 (重要)**: 显著提升匹配精度和用户粘性，是差异化竞争的关键
- **P2 (可延迟)**: 增强体验或架构优化，可在P0/P1之后迭代

### F5: 需求发布市场（P0 🟢已开发80%）

#### 3.1 功能概述

允许企业用户在链客宝平台上主动发布**资源/合作/采购需求**，匹配引擎自动推荐合适的企业（供给方），同时也支持供给方发现并响应需求。将链客宝从「被动匹配」升级为「主动发布 + 被动匹配」双侧驱动。

#### 3.2 B2B化改造要点

- 原始询赋模型面向个人（技能/私单）→ 改造为面向企业（资源/采购/合作）
- 需求类型：供应链采购、渠道合作、技术外包、投资融资、联合研发、场地/设备共享
- 匹配目标：企业→企业（非个人→个人）

#### 3.3 用户故事

| 角色 | 用户故事 |
|:----|:---------|
| **需求方企业（采购方）** | 作为一家需要采购原材料的韩国中小企业主，我希望在链客宝发布「寻找优质钢材供应商」的需求，平台自动为我推荐3-5家合适的供应商，我可以直接发起洽谈 |
| **供给方企业（产品方）** | 作为一家拥有优质钢材库存的企业，我希望看到平台上与企业产品匹配的需求，主动响应并报价，拓展新客户 |
| **渠道方企业（推广员）** | 作为拥有行业人脉的渠道商，我希望看到平台上双方发布的供需信息，撮合双方交易并赚取分润 |
| **平台运营方** | 作为平台管理员，我希望审核需求内容的合规性，确保信息真实可靠 |

#### 3.4 功能需求

##### 3.4.1 需求发布

| ID | 需求 | 说明 |
|:--:|:----|:-----|
| RQ-F5-01 | 需求发布表单 | 支持填写：需求标题、需求类型（下拉选择）、详细描述、预算范围（选填）、期望合作方式、截止日期、附件上传 |
| RQ-F5-02 | 需求类型分类 | 预设8-10种B2B需求类型：产品采购/技术外包/渠道合作/投资融资/联合研发/代工生产/物流仓储/场地共享 |
| RQ-F5-03 | 需求可见性设置 | 公开（所有人可见）/ 定向（仅指定企业可见）/ 内部（仅企业成员可见） |
| RQ-F5-04 | AI辅助填写 | 输入关键词后，AI自动生成需求描述草稿并推荐匹配行业标签 |
| RQ-F5-05 | 需求审核机制 | 新发布需求需经平台审核（自动NLP过滤违规内容 + 人工抽查），审核通过前仅发布企业可见 |

##### 3.4.2 需求大厅

| ID | 需求 | 说明 |
|:--:|:----|:-----|
| RQ-F5-06 | 需求列表页 | 按时间倒序展示公开需求，支持分类筛选、关键词搜索、排序（最新/最热/即将截止） |
| RQ-F5-07 | 需求详情页 | 展示完整需求信息 + 发布企业基本信息（非敏感部分）+ 响应按钮 + 收藏功能 |
| RQ-F5-08 | 智能推荐 | 匹配引擎根据企业画像向供给方推荐相关需求（Bandit算法+Thompson sampling） |
| RQ-F5-09 | 需求动态通知 | 发布企业收到响应/收藏/匹配通知；供给方收到新匹配需求通知 |

##### 3.4.3 需求响应

| ID | 需求 | 说明 |
|:--:|:----|:-----|
| RQ-F5-10 | 企业响应需求 | 供给方点击「响应需求」，填写：合作提案（文字）、报价（选填）、相关案例（选填） |
| RQ-F5-11 | 需求方确认 | 需求方查看所有响应，筛选合适合作方，发起洽谈 |
| RQ-F5-12 | 需求状态流转 | 草稿→待审核→审核失败→已发布→响应中→洽谈中→已完成→已关闭 |
| RQ-F5-13 | 需求关闭/下架 | 发布企业可手动关闭需求，或在截止日期后自动关闭 |

#### 3.5 验收标准

| ID | 标准 | 验证方式 |
|:--:|:----|:---------|
| AC-F5-01 | 企业用户可完成需求发布全流程（表单→提交→审核→发布） | 端到端测试 |
| AC-F5-02 | 需求大厅展示至少5种需求类型的筛选结果 | 功能测试 |
| AC-F5-03 | 匹配引擎在需求发布后30秒内向相关供给方推送通知 | 延迟测试 |
| AC-F5-04 | 供给方可在需求详情页发起响应，需求方收到响应通知 | 端到端测试 |
| AC-F5-05 | 审核系统自动拦截含敏感词/违规内容的需求 | NLP测试 |
| AC-F5-06 | 响应数据不得泄露给非相关方（数据隔离） | 安全审计 |

#### 3.6 技术建议

- **数据模型**: 新建 `Demand` 表（id, company_id, type, title, description, budget, status, visibility, deadline, created_at, updated_at）+ `DemandResponse` 表（id, demand_id, company_id, proposal, quote, status）
- **匹配扩展**: 在现有三塔DNN基础上增加「需求-企业匹配塔」，输入为需求embedding + 企业embedding，输出匹配分数。与现有产品-企业匹配共享底层embedding层
- **审核机制**: 先NLP关键词过滤（敏感词库），再AI判断（LLM评估），最后人工抽查（10%样本）
- **数据安全**: 需求数据默认只对发布企业和被匹配企业可见；匹配过程在企业内网完成（韩国数据不离境）；需求详情页不展示企业敏感信息（联系方式等需响应后解锁）

#### 3.7 数据安全考虑

- 需求内容加密存储（AES-256），仅授权用户可解密查看
- 匹配过程全部在企业内网完成，不经过第三方云服务
- 需求数据保留期限可配置（默认12个月，到期自动归档）
- 企业联系人信息仅在双方确认合作意向后披露（双向同意机制）

### F1: 多身份角色系统（P1 🔴缺失 30%）

#### 3.8 功能概述

允许企业以**不同角色身份**在链客宝平台上展示不同的能力和资源。一个企业可以同时是「产品方」「采购方」「渠道方」「服务方」，每种身份有独立的展示页面和匹配优先级。

#### 3.9 B2B化改造要点

- 原始询赋面向个人（职场人/自由职业者/微商）→ 改造为面向企业的业务角色
- 企业角色预置：`product_provider`（产品供给方）、`procurement`（采购需求方）、`channel_partner`（渠道推广方）、`service_provider`（服务提供方）、`investor`（投资方）
- 不等于用户角色（如C端后台角色admin/editor），而是**企业的业务身份**

#### 3.10 用户故事

| 角色 | 用户故事 |
|:----|:---------|
| **综合型企业** | 作为一家同时有产品销售和采购需求的企业，我希望为「产品方」身份展示产品目录，为「采购方」身份展示采购需求，两种身份独立匹配不同的合作方 |
| **渠道型企业** | 作为一家以渠道推广为核心业务的企业，我希望启用「渠道方」身份，展示可推广的渠道资源和历史业绩，匹配适合的产品方 |
| **平台运营方** | 作为平台管理员，我希望看到企业启用的所有角色，根据角色分配不同的平台功能权限 |

#### 3.11 功能需求

| ID | 需求 | 说明 |
|:--:|:----|:-----|
| RQ-F1-01 | 角色管理界面 | 企业可在「企业设置」中启用/禁用角色，每个角色可独立编辑名称、简介、标签、能力描述 |
| RQ-F1-02 | 预置角色模板 | 5个预置角色（产品方/采购方/渠道方/服务方/投资方），每个角色有默认字段和填写引导 |
| RQ-F1-03 | 角色可见性 | 企业可设置每个角色是否对外展示、匹配优先级排序 |
| RQ-F1-04 | 角色切换 | 在首页/个人中心支持角色切换，不同角色看到不同的推荐内容和数据看板 |
| RQ-F1-05 | 角色匹配权重 | 匹配引擎将角色的「标签匹配度」作为重要输入特征（权重可配置） |

#### 3.12 验收标准

| ID | 标准 | 验证方式 |
|:--:|:----|:---------|
| AC-F1-01 | 企业可启用至少3种角色并独立编辑信息 | 功能测试 |
| AC-F1-02 | 角色切换后，匹配推荐内容和数据看板不同 | 端到端测试 |
| AC-F1-03 | 匹配结果可根据角色粒度展示匹配理由 | UI测试 |
| AC-F1-04 | 角色信息在API响应中正确返回 | API测试 |

#### 3.13 技术建议

- **数据模型**: 在 `Company` 表增加 `roles` 字段（JSON数组），新建 `CompanyRole` 表（id, company_id, role_type, name, description, tags, status, priority）
- **匹配引擎修改**: 三塔DNN的用户塔改为「企业塔 + 角色塔」双塔结构，角色塔输入为角色embedding
- **前端改动**: 用户中心增加角色管理页面；首页增加角色切换器组件

#### 3.14 数据安全考虑

- 角色信息属于企业可配置的公开信息，由企业自主控制可见性
- 未启用的角色数据不对外暴露
- 角色切换行为和偏好数据不跨境传输

### F3: 社交信任链权重匹配（P1 🟢已开发80%）

#### 3.15 功能概述

在匹配引擎中引入**B2B社交信任维度**，将企业间的已有关系（共同合作方、上下游关系、同园区、同行业联盟）作为匹配权重因子。匹配结果中，有信任链连接的企业优先展示。

#### 3.16 B2B化改造要点

- 原始询赋是C端社交图谱（好友的好友）→ 改造为B2B关系图谱（共同合作企业/上下游/同园区/同商会）
- 不使用个人社交关系（微信/电话通讯录），只使用**公开的B2B关系数据**
- 信任链权重作为Bandit算法的context特征，而非替代核心匹配逻辑

#### 3.17 用户故事

| 角色 | 用户故事 |
|:----|:---------|
| **制造型企业** | 作为一家制造企业，我信任我的供应商推荐的合作方，因为供应商已与我建立长期合作关系。匹配结果中，与我有共同供应商的企业应优先展示 |
| **园区企业** | 作为一家入驻韩国产业园区的企业，我希望优先匹配同在园区的企业，方便线下交流和降低信任成本 |
| **行业联盟企业** | 作为一名商会会员企业，我希望匹配结果中优先展示同商会会员企业 |

#### 3.18 功能需求

| ID | 需求 | 说明 |
|:--:|:----|:-----|
| RQ-F3-01 | B2B关系图谱构建 | 基于已有企业间合作记录、同园区、同行业标签等构建企业关系图谱（图数据库） |
| RQ-F3-02 | 共同合作方匹配 | 匹配引擎检查两家企业是否有共同合作方，有则加分（权重可配置） |
| RQ-F3-03 | 同园区/同区域权重 | 同园区/同城市/同国家的企业在匹配结果中优先展示（距离衰减模型） |
| RQ-F3-04 | 上下游关系推荐 | 基于企业品类标签，识别潜在上下游关系（如原材料→加工厂→批发商→零售商） |
| RQ-F3-05 | 信任链可视化 | 匹配结果页展示「信任链」标签（如：你们都有共同合作方「XX公司」） |
| RQ-F3-06 | 信任链隐私保护 | 不展示完整的社交关系图，仅展示「信任链」存在性标签 |
| RQ-F3-07 | 权重可配置 | 平台管理员可配置各维度权重（初始建议：共同合作方30% / 同园区20% / 上下游20% / 同行业30%） |

#### 3.19 验收标准

| ID | 标准 | 验证方式 |
|:--:|:----|:---------|
| AC-F3-01 | 两家有共同合作方的企业，匹配分数高于无关系企业 | A/B测试 |
| AC-F3-02 | 同园区企业匹配分数高于非同园区企业 | A/B测试 |
| AC-F3-03 | 匹配结果展示「信任链」标签 | UI测试 |
| AC-F3-04 | 信任链权重加入Bandit算法后，CTR提升≥5% | 在线实验 |
| AC-F3-05 | 企业关系数据不对外暴露，仅用于匹配权重计算 | 安全审计 |

#### 3.20 技术建议

- **图数据库**: 使用Neo4j或ArangoDB存储企业关系图谱（关系类型：合作、同园区、同联盟、上下游）
- **匹配引擎集成**: 在Bandit算法的context特征中加入 `trust_chain_score` 特征，作为排序层的输入。不修改召回逻辑，只在重排阶段加权
- **关系数据来源**: ① 平台内合作记录（自动生成） ② 企业填写的园区/联盟信息 ③ 行业分类推断的上下游关系
- **隐私设计**: 信任链权重计算在服务端完成，不对客户端暴露关系图谱细节

#### 3.21 数据安全考虑

- 企业关系数据属于衍生数据，不包含个人隐私信息
- 关系图在企业内网构建，不上传任何第三方
- 「信任链」仅展示存在性标签（如「有1个共同合作方」），不展示具体是谁
- 企业可选择退出信任链匹配（opt-out）

### F8: 批量导入建联（P1 🟢已开发90%）

#### 3.22 功能概述

支持企业用户以**CSV/Excel格式批量导入**合作伙伴/供应商/客户的企业信息，系统自动匹配已有企业数据、识别重复项，批量建立企业连接。

#### 3.23 B2B化改造要点

- 原始询赋面向个人通讯录/微信好友导入 → 改造为面向企业数据的CSV/Excel批量导入
- 重点：智能匹配去重（同一家企业不同写法如「三星电子」vs「Samsung Electronics」自动识别）
- 不导入个人微信/通讯录数据（B2B不依赖C端社交关系）

#### 3.24 用户故事

| 角色 | 用户故事 |
|:----|:---------|
| **外贸企业** | 作为一家有200家客户信息的贸易企业，我不想逐个手动添加。我希望能上传一份Excel表格，系统自动识别已有企业、去重、批量建立连接 |
| **展会参与企业** | 作为参加完展会的企业，我收集了50张名片，希望拍照导入或按模板录入，批量添加到我的企业网络中 |
| **平台运营方** | 作为平台运营，我希望企业批量导入时自动补充公开信息（如企业工商信息），提高数据完整度 |

#### 3.25 功能需求

| ID | 需求 | 说明 |
|:--:|:----|:-----|
| RQ-F8-01 | CSV/Excel上传 | 支持上传CSV和Excel格式文件（含示例模板下载），最大支持5000行 |
| RQ-F8-02 | 导入模板 | 提供标准模板（至少有：企业名称、联系人、电话、邮箱、行业），可选字段（地址、网址、简介） |
| RQ-F8-03 | 智能去重 | 自动识别已存在于平台的企业，提示合并还是新建连接 |
| RQ-F8-04 | 企业名称标准化 | 模糊匹配企业名称（如「三星星」→「三星电子」），减少重复录入 |
| RQ-F8-05 | 批量建联 | 导入成功后，系统自动为这些企业建立「连接」关系（非好友，而是B2B企业连接） |
| RQ-F8-06 | 导入结果报告 | 导入完成后展示处理结果：成功数/重复数/失败数/失败原因 |
| RQ-F8-07 | 手动录入 | 支持单条手动录入企业信息（作为补充） |

#### 3.26 验收标准

| ID | 标准 | 验证方式 |
|:--:|:----|:---------|
| AC-F8-01 | 上传1000行CSV文件，5秒内完成解析和导入 | 性能测试 |
| AC-F8-02 | 同一企业名不同写法（大小写/简繁体）自动识别为重复 | 模糊匹配测试 |
| AC-F8-03 | 导入结果报告正确显示成功/重复/失败数量 | 功能测试 |
| AC-F8-04 | 导入后企业连接关系在5分钟内生效 | 端到端测试 |

#### 3.27 技术建议

- **解析引擎**: 使用 `pandas` + `openpyxl` 解析Excel；使用Python `csv` 模块解析CSV
- **模糊匹配**: 使用 `fuzzywuzzy` 或 `rapidfuzz` 进行企业名称模糊匹配（阈值≥85%标记为重复）
- **批量处理**: 使用 `asyncio` 批量处理+进度条显示；超过5000行分批次处理
- **数据模型**: 在现有 `Company` 表加 `batch_import_id` 字段记录批量导入来源

#### 3.28 数据安全考虑

- 导入数据仅在企业内部可见，不跨企业共享
- 导入的Excel/CSV文件处理完成后自动清除（不保存原始文件）
- 企业信息匹配去重过程在企业内网完成
- 导入失败的数据不日志记录（仅记录失败数和原因类型）

| **F7: 匹配协议标准化** | 🟢 低 — 纯架构改造 | 🔴 高 — 涉及匹配引擎核心重构 | 🟢 低 — 对用户透明 | **中高** | 🔴 10%未开始 |

### 5.2 跨Feature风险矩阵（v1.1 审计更新）

| 风险 | 影响Feature | 当前状态 | 缓解措施 |
|:-----|:----------|:---------|:---------|
| **匹配引擎改造冲突** | F1 + F3 + F5 | F3/F5增强匹配已开发完成，F1未动 | 统一规划匹配引擎v2.0架构，F1的匹配修改在单独Sprint完成 |

| **用户冷启动** | F5（需求发布） | BusinessNeed模型已有，seed_needs.py种子数据脚本存在 | 发布内置「种子需求」（平台引导发布），首月运营激励机制 |

### 5.3 关键风险与应对建议

#### 5.3.1 首要风险: 匹配引擎稳定性

**风险**: F1+F3+F5三个Feature同时修改匹配引擎（三塔DNN+Bandit），可能影响现有匹配准确率和系统稳定性。

**应对**:
1. 所有匹配引擎修改均通过A/B测试验证（对照+实验组），CTR下降超过3%立刻回滚
2. 新增匹配维度先以「可选开关」形式加入，默认关闭，逐步放开
3. 保留旧版匹配引擎作为fallback

#### 5.3.2 次要风险: 数据安全合规

**风险**: 韩国企业数据不离开内网的要求与社交图谱构建可能冲突。

**应对**:
1. 关系图谱完全在企业内网构建和存储
2. 置信链权重计算离线完成，结果缓存到内网数据库
3. 批量导入的原始文件处理完后立即清除

#### 5.3.3 机会风险: 用户习惯改变

**风险**: 需求发布市场和多身份角色系统改变了用户使用链客宝的方式，需要教育成本。

**应对**:
1. F5上线时配套「三阶段用户引导」（粘贴运行→项目驱动→最佳实践）
2. F1上线时提供行业最佳实践模板（预设角色配置）
3. 监控用户角色切换率和需求发布量，低于预期则增加运营激励
