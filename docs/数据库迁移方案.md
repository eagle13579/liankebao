# 链客宝 数据库迁移方案

> 适用版本：v1.0+
> 最后更新：2026-05-27

---

## 一、概述

链客宝项目默认使用 SQLite 作为开发/单机数据库，支持通过环境变量切换为 MySQL 或 PostgreSQL。本文档详细说明 **SQLite → MySQL** 的迁移路径、表结构映射关系、备份策略与回滚方案。

### 迁移目标

| 项目 | 说明 |
|------|------|
| **源数据库** | SQLite (`chainke.db`，默认路径 `backend/data/`) |
| **目标数据库** | MySQL 8.0+（InnoDB 引擎，utf8mb4 字符集） |
| **迁移工具** | Python SQLAlchemy（ORM级迁移）+ 可选 SQL 脚本 |
| **数据量级** | 小型-中型（< 10万条记录） |

## 二、迁移路径

### 2.1 路径总览

```
┌──────────────┐     ┌──────────────────┐     ┌──────────────────┐
│   SQLite     │ ──▶ │ 数据导出 & 校验   │ ──▶ │   MySQL 8.0+     │
│  chainke.db  │     │  (Python 脚本)    │     │  chainke 数据库   │
└──────────────┘     └──────────────────┘     └──────────────────┘
                              │
                              ▼
                     ┌──────────────────┐
                     │  SQL 脚本方式     │
                     │  V001 + V002     │
                     │  (兜底/CI方案)    │
                     └──────────────────┘
```

### 2.2 方式一：Python 一键迁移（推荐）

使用项目现有的 `one_click_migrate.py` 脚本，支持 4 步自动化流程：

```bash
# 1. 设置环境变量
export DB_TYPE=mysql
export DATABASE_URL=mysql+pymysql://user:password@localhost:3306/chainke?charset=utf8mb4

# 2. 执行迁移
cd backend
python scripts/one_click_migrate.py --to mysql -y
```

流程内包含：
1. **Step 1** — 数据源验证（检查表结构、行数）
2. **Step 2** — 创建 MySQL 表结构（通过 SQLAlchemy ORM `metadata.create_all`）
3. **Step 3** — 数据批量写入（每批 100 条，含事务回滚）
4. **Step 4** — 数据一致性校验（逐表对比行数，给出差异报告）

### 2.3 方式二：SQL 脚本迁移（兜底方案）

用于 MySQL CLI 直连、CI/CD 自动化、或 Python 环境不可用的场景：

```bash
# 1. 在 MySQL 中创建数据库
mysql -u root -p -e "CREATE DATABASE IF NOT EXISTS chainke CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"

# 2. 按版本号顺序执行 SQL
cd deploy/migrations
mysql -u root -p chainke < V001__initial_schema.sql
mysql -u root -p chainke < V002__seed_data.sql

# 或使用 migrate.sh 一键执行
chmod +x migrate.sh
./migrate.sh -u root -p'your_password' -d chainke
```

### 2.4 方式三：零停机迁移（生产环境）

```bash
# 1. 先导出 SQLite 数据为 SQL
sqlite3 backend/data/chainke.db .dump > /tmp/sqlite_dump.sql

# 2. 启动 MySQL 目标库
# 3. 使用 V001 创建表结构
mysql -u root -p chainke < deploy/migrations/V001__initial_schema.sql

# 4. 使用迁移脚本写入数据（不清空目标表）
cd backend
python scripts/migrate_to_mysql.py --no-truncate -y

# 5. 切换环境变量后重启服务
export DB_TYPE=mysql
export DATABASE_URL=mysql+pymysql://user:password@localhost:3306/chainke?charset=utf8mb4
# 重启 FastAPI 服务
```

## 三、表结构映射

### 3.1 数据库类型差异对照

| 特性 | SQLite | MySQL |
|------|--------|-------|
| 整型自增 | `INTEGER PRIMARY KEY AUTOINCREMENT` | `INT AUTO_INCREMENT` |
| 浮点数 | `REAL` / `FLOAT` | `FLOAT` / `DOUBLE` |
| 定点数 | 无原生支持（存为 REAL） | `DECIMAL(12,2)` |
| 字符串 | `VARCHAR` 等效 TEXT | `VARCHAR(N)` |
| 大文本 | `TEXT` | `TEXT` / `LONGTEXT` |
| 日期时间 | `DATETIME` | `DATETIME` |
| JSON | 存为 TEXT | 可用 `JSON` 类型（SQL 脚本中使用 TEXT 兼容） |
| 布尔 | 0/1（INTEGER） | 0/1（TINYINT） |
| 引擎 | 无 | 使用 `ENGINE=InnoDB` |
| 字符集 | 无 | `DEFAULT CHARSET=utf8mb4 COLLATE=utf8mb4_unicode_ci` |
| 外键约束 | 需 `PRAGMA foreign_keys = ON` | 默认启用 |
| 索引 | 支持的 B-tree | 支持的 B-tree + 全文索引 |

## 四、备份策略

### 4.1 迁移前备份（必做）

#### SQLite 备份
```bash
# 方式一：文件复制（推荐）
cp backend/data/chainke.db backend/data/chainke.db.bak.$(date +%Y%m%d_%H%M%S)

# 方式二：使用 sqlite3 备份（避免锁表）
sqlite3 backend/data/chainke.db ".backup 'backend/data/chainke_pre_migrate.bak'"

# 方式三：导出为 SQL 文件
sqlite3 backend/data/chainke.db ".dump" > /tmp/chainke_pre_migrate_dump.sql
```

#### MySQL 备份（如果已有数据）
```bash
mysqldump -u root -p --databases chainke > /tmp/chainke_mysql_bak_$(date +%Y%m%d).sql
```

### 4.2 迁移后备份
```bash
# 迁移完成后，立即对 MySQL 做全量备份
mysqldump -u root -p --routines --triggers --databases chainke > /tmp/chainke_mysql_after_migrate.sql
```

### 4.3 定期备份建议
| 频率 | 方式 | 保留策略 |
|------|------|----------|
| 每日 | mysqldump 全量 | 保留 7 天 |
| 每周 | mysqldump + gzip | 保留 4 周 |
| 每月 | mysqldump + gzip，异地存储 | 保留 12 个月 |

## 五、回滚方案

### 5.1 回滚判断标准

当以下任一情况出现时，应执行回滚：

1. **数据一致性校验失败**：SQLite 与 MySQL 行数不匹配
2. **业务功能异常**：迁移后核心 API（登录、下单、支付）报错
3. **性能下降**：查询响应时间超过迁移前的 2 倍
4. **外键约束违反**：数据关联关系丢失

### 5.2 回滚步骤

#### 第一阶段：停止服务
```bash
# 停止当前 FastAPI 服务（假设使用 systemd）
sudo systemctl stop chainke

# 或直接 kill 进程
pkill -f "uvicorn main:app"
```

#### 第二阶段：切回 SQLite
```bash
# 1. 恢复 SQLite 备份
cp backend/data/chainke.db.bak.20260527_120000 backend/data/chainke.db

# 2. 确认备份完整性
sqlite3 backend/data/chainke.db "SELECT COUNT(*) FROM users;"

# 3. 修改环境变量（如果已设置 MySQL）
unset DB_TYPE
unset DATABASE_URL

# 或直接设置为 SQLite
export DB_TYPE=sqlite
```

#### 第三阶段：重启并验证
```bash
# 启动服务
sudo systemctl start chainke

# 或
cd backend && nohup python main.py &

# 验证：
# - 访问 http://localhost:8000/docs 确认 Swagger 正常
# - 测试登录 /api/auth/login
# - 检查首页数据是否正确渲染
```

#### 第四阶段：清理 MySQL 数据（可选）
```bash
# 如果 MySQL 中写入不完整的数据需要清理
mysql -u root -p chainke -e "DROP DATABASE chainke; CREATE DATABASE chainke CHARACTER SET utf8mb4 COLLATE utf8mb4_unicode_ci;"
```

### 5.3 自动回滚（迁移脚本内置）

`one_click_migrate.py` 和 `migrate_to_mysql.py` 在以下情况会自动事务回滚：

- 数据写入异常 → 当前表事务 `rollback()`，抛出异常
- 校验不通过 → 输出差异报告，**不会自动清理 MySQL 数据**（需手动处理）

> ⚠️ **重要**：迁移脚本不会自动删除已写入的 MySQL 数据。如需清空，使用 `TRUNCATE` 或 `DROP DATABASE`。

## 六、校验机制

### 6.1 迁移前校验（pre_validate）

| 检查项 | 说明 |
|--------|------|
| 表存在性 | 检查所有必需表是否存在 |
| 列完整性 | 检查关键列是否存在 |
| 行数统计 | 记录迁移前各表行数 |
| 空数据警告 | 空表给出警告 |

### 6.2 迁移后校验（verify_migration）

| 检查项 | 说明 |
|--------|------|
| 行数对比 | SQLite vs MySQL 逐表对比 |
| 差异报告 | 给出差异数量和差异表名 |
| 票决结果 | all_match = true/false |

### 6.3 手动校验 SQL

```sql
-- 用户表
SELECT 'users' AS table_name, COUNT(*) AS row_count FROM users;
-- 产品表
SELECT 'products' AS table_name, COUNT(*) AS row_count FROM products;
-- 订单表
SELECT 'orders' AS table_name, COUNT(*) AS row_count FROM orders;
-- 联系人表
SELECT 'contacts' AS table_name, COUNT(*) AS row_count FROM contacts;
-- 提现表
SELECT 'withdrawals' AS table_name, COUNT(*) AS row_count FROM withdrawals;
-- 余额表
SELECT 'user_balances' AS table_name, COUNT(*) AS row_count FROM user_balances;
```

## 八、FAQ

### Q: 迁移后需要修改代码吗？

不需要。项目使用 SQLAlchemy ORM，通过 `DB_TYPE` 环境变量自动适配。迁移后只需设置 `DB_TYPE=mysql` 和 `DATABASE_URL` 即可。

### Q: SQLite 中的 JSON 字段如何处理？

SQLite 将 JSON 存储为 TEXT 字符串，迁移到 MySQL 后同样使用 TEXT 类型存储。应用层通过 `json.loads()` / `json.dumps()` 序列化/反序列化，无需额外处理。

### Q: 浮点数精度问题

SQLite 使用 `FLOAT`/`REAL`，MySQL 同样使用 `FLOAT`。对于金额相关字段（如余额表），MySQL 使用 `DECIMAL(12,2)` 确保精确。迁移脚本通过 SQLAlchemy ORM 自动处理类型映射。

### Q: 迁移耗时多久？

数据量 < 1万条时通常 < 30 秒。主要耗时在 MySQL 端写入速度。可通过调整 `batch_size`（默认 100）优化。

### Q: 迁移后 SQLite 可以继续使用吗？

可以。但建议在确认 MySQL 稳定运行后，将 SQLite 数据库文件归档或删除，避免混淆。
