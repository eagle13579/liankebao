## 1. 概述

### 1.1 定位

L2 调度层位于 L1 接入层（统一网关）和 L3 执行层（AI 子代理委派）之间，是**供需匹配的核心编排中枢**。负责：

- 接收需求方发布的需求，写入匹配队列
- 管理需求的全生命周期状态（状态机）
- 触发 L3 执行层做 AI 评分匹配
- 协调多角色接力（系统→运营→双方洽谈）
- 对外暴露 Kanban 风格的视图

### 1.2 当前问题

| 问题 | 现状 | 影响 |
|------|------|------|
| 无匹配工作流 | 仅有基础 CRUD，无状态流转 | 供需双方失联，成交率低 |
| 无接力机制 | 需求发布后无人跟进 | 需求沉底，响应慢 |
| 无任务状态追踪 | 无法知道需求当前在哪个环节 | 运营不可控，数据不可见 |

### 1.3 设计目标

1. **需求发布标准化** — 统一 API 规范，版本兼容
2. **匹配队列自动化** — 发布即入队，自动触发匹配
3. **状态机严谨** — 五态流转，无非法跳转
4. **多角色可协作** — 系统·运营·双方各有职责

## 4. 匹配队列状态机

### 4.1 状态定义

```
┌─────────┐     ┌──────────┐     ┌────────────┐     ┌─────────────┐
│ pending │────▶│processing│────▶│recommended │────▶│negotiating  │
│ (待匹配) │     │ (匹配中)  │     │ (已推荐)    │     │ (洽谈中)     │
└────┬────┘     └──────────┘     └─────┬──────┘     └──────┬──────┘
     │                                 │                   │
     │ 超时                             │ 超时              │ 超时
     ▼                                 ▼                   ▼
┌─────────┐                    ┌──────────┐       ┌──────────┐
│ expired │                    │ expired  │       │ expired  │
│ (已过期) │                    │ (已过期)  │       │ (已过期)  │
└─────────┘                    └──────────┘       └──────────┘

     ┌────────────┐
     │ completed │  ← 从 negotiating 达成一致
     │ (已成交)   │
     └────────────┘
```

### 4.2 状态转移矩阵

| 当前 \\ 目标 | pending | processing | recommended | negotiating | completed | expired |
|:---:|:---:|:---:|:---:|:---:|:---:|:---:|
| pending | - | ✅ | ❌ | ❌ | ❌ | ✅ |
| processing | ❌ | - | ✅ | ❌ | ❌ | ✅ |
| recommended | ❌ | ❌ | - | ✅ | ❌ | ✅ |
| negotiating | ❌ | ❌ | ❌ | - | ✅ | ✅ |
| completed | ❌ | ❌ | ❌ | ❌ | - | ❌ |
| expired | ❌ | ❌ | ❌ | ❌ | ❌ | - |

**说明：**
- `pending → processing`：入队后自动触发，也可以手动重试
- `processing → recommended`：L3 执行层返回结果后自动推进
- `recommended → negotiating`：运营或需求方确认推荐、开始洽谈
- `negotiating → completed`：双方达成合作意向/成交
- `* → expired`：各状态超时后自动过期（超时阈值见 4.4）
- 已成交/已过期的状态不可再变更

### 4.3 状态触发者

| 转移 | 触发者 | 触发条件 |
|------|--------|----------|
| pending → processing | system | 发布后立即触发 / 定时轮询出队 |
| processing → recommended | system | L3 执行层回调 / 轮询检测完成 |
| recommended → negotiating | operator / demand_owner | 运营确认 / 需求方主动开始洽谈 |
| negotiating → completed | operator | 双方确认成交 |
| * → expired | system | 超时定时任务（每 5 分钟扫描） |

### 4.4 超时策略

| 状态 | 最大停留时间 | 超时后动作 |
|------|------------|-----------|
| pending | 30 分钟 | → expired |
| processing | 10 分钟 | 重试 3 次 → 标记为 failed（人工介入） |
| recommended | 48 小时 | → expired（发送提醒给运营） |
| negotiating | 7 天 | → expired（发送提醒给双方） |

## 5. 队列调度策略

### 5.1 队列数据结构

使用 Redis 列表（List）或 数据库队列表：

```
键名: demand_queue:pending     (List: 待匹配需求 ID)
键名: demand_queue:processing  (List: 匹配中需求 ID)
键名: demand_queue:recommended (List: 已推荐需求 ID)
```

### 5.2 出队策略

| 维度 | 策略 |
|------|------|
| 优先级 | high > normal > low（按 urgency 字段排序）|
| 时间 | 同优先级内 FIFO（先进先出）|
| 批量 | 每次出队 1 个需求，递交给 L3 执行层 |
| 并发控制 | 最多同时处理 5 个需求（可配置）|
| 超时回滚 | processing 超过 10 分钟未完成的，重试或回退到 pending |

### 5.3 调度定时器

| 任务 | 频率 | 说明 |
|------|------|------|
| 需求出队调度 | 每 30 秒 | 从 pending_queue 取 1 个→processing→调用 L3 |
| 超时扫描 | 每 5 分钟 | 检查各状态超时→自动 expired |
| 重试检查 | 每 2 分钟 | 检查 failed 状态任务→重新入队（最多 3 次）|

## 6. 多角色接力机制

### 6.1 角色定义

| 角色 | 职责 | 可操作的状态转移 |
|------|------|----------------|
| system | AI 自动处理 | pending→processing, processing→recommended, *→expired |
| operator | 运营人员确认 | recommended→negotiating |
| demand_owner | 需求方 | 查看状态/确认推荐结果 |
| supply_owner | 供应方 | 查看状态/接受合作 |

### 6.2 接力流程

```
Step 1: system → 需求自动匹配评分 (L3 执行)
Step 2: operator → 人工复核推荐结果，调整评分/添加备注
Step 3: demand_owner → 查看推荐列表，选择意向供应商
Step 4: demand_owner + supply_owner → 双方洽谈（线下/平台 IM）
Step 5: operator → 确认成交，完成闭环
```
