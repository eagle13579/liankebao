## 各成员学习收获

### 白泽 (CEO)
1. **代码阶段比文档阶段更需要精确的路径传递** — delegate_task context 中写死 `D:\\chainke-full\\deploy\\docker\\` 和 `D:\\链客宝\\backend\\app\\` 是代码迁移成功的关键。路径错了就全错。
2. **npm install 在 deploy/docker/ 下而非根目录** — 项目的node_modules结构特殊(vite配置在deploy/docker/, 源文件在根目录src/)。给子Agent的context必须显式注明工作目录。
3. **Python语法验证要事后统一做** — 3路子Agent都声称"语法通过"但CEO必须统一验证。本次4个.py文件全部通过ast.parse，没有遗漏。

### 山膏 (视觉P8) — Tailwind基建
1. **tailwindcss v4 使用 `@import "tailwindcss"` 而非旧版的三层指令** — @tailwind base/components/utilities 已废弃，单行导入即可。
2. **@theme 定义的是CSS变量, 不是Tailwind class名** — `--color-brand-500: #3b82f6` 定义后, class名是 `bg-brand-500` 而非 `bg-blue-500`。需要组件层逐一迁移。
3. **vite-plugin 在v6中更名为 @tailwindcss/vite** — 旧版 tailwindcss/plugin 不再兼容，必须用新包名。

### 烛龙 (工程P8) — ORM模型 + auth适配
1. **chainke-full 用 String(64) 而非 ForeignKey 关联用户** — 这是架构级差异。4个模型全部改用 String, 无数据库级外键约束。
2. **AuthMiddleware 注入 request.state.user, 源项目用 Depends(get_current_user)** — 适配层 `auth.py` 桥接了两种认证模式, 迁移后可同时兼容。
3. **__init__.py 的 __all__ 必须同步更新** — 追加模型 import 后, __all__ 列表若不更新, `from app.models import *` 会遗漏。

### 文鳐 (助理P9) — API Key中间件 + Webhook
1. **api_key_auth 依赖 developer_portal_models — 创建顺序必须模型先行** — 中间件 import developer_portal_models 中的 ApiKey/ApiUsageLog, 必须先建模型再注册中间件。
2. **webhook_v2.py 发现源项目的一个bug** — `datetime.now(timezone.utcnow())` → 应为 `datetime.now(timezone.utc)`, 源项目代码写错参数名。迁移时顺手修复。
3. **纯内存webhook引擎+文件持久化** — chainke-full 的 webhook_v2 使用纯内存运行, 无 DB 依赖, 启动快但重启后丢失订阅。适合 MVP 阶段。
