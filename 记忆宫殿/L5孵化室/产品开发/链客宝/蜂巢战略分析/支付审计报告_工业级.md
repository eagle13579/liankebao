```
# 模式开关

# 可选
WXPAY_REFUND_NOTIFY_URL=https://api.example.com/api/payment/wxpay/refund-callback
WXPAY_ROOT_CA_PATH=/etc/certs/rootca.pem   # 微信根证书
SERVER_IP=1.2.3.4                           # 服务器公网IP（V2统一下单用）
```

**获取证书序列号命令**:

## 五、关于payjs.cn

**审计结论：当前代码完全不用 payjs.cn。**

现有代码两种方式都是直连微信官方API：
- `wechat_pay.py`（V2）：直接调用 `https://api.mch.weixin.qq.com/pay/unifiedorder`
- `payment/wxpay/`（V3）：直接调用 `https://api.mch.weixin.qq.com/v3/pay/transactions/jsapi`

**不是 payjs.cn 聚合支付，是直连微信。**

如果你之前用过 payjs.cn（第三方聚合支付，帮你收钱然后再结算给你），那么当前代码已经"跳过"了 payjs，直接对接了微信官方。payjs 的好处是门槛低（个体工商户即可），但费率更高（0.38%起 vs 微信官方0.6%但更稳定）。

## 七、关键发现

### 1. 后端入口注册顺序
`main.py` 中 `ensure_payment_config()` 在模块导入时执行（payment.py 第78行），早于FastAPI start事件。这意味着支付配置在路由注册前就已加载。**正确。**

### 2. 数据库表创建
三张充值相关表（`user_balances`, `recharge_orders`, `balance_logs`）在 `recharge/models.py` 中定义，但需要确保它们通过 SQLAlchemy Base 自动创建（`init_db()` 调用了 `Base.metadata.create_all()`）。当前主模型文件（`app/models.py`）中不包含这些表——它们是独立模块。确认 `main.py` 中导入了 `recharge.routes` 和 `recharge.callback`，这会触发 `recharge/__init__.py` 和 `recharge/models.py` 的导入，从而注册表。**正确。**

### 3. User模型有 wechat_openid 字段
`User.wechat_openid` 已存在（`app/models.py` 第16行）。充值路由依赖此字段作为微信支付的 openid 来源。**正确。**

### 4. 环境变量兼容
`payment/config.py` 同时支持 `WXPAY_*` 和 `WECHAT_*` 两种前缀，且兼容 `APPID`/`APP_ID`、`MCH_ID`/`MCHID` 等变体名。这是为避免老配置不兼容而设计的。**优秀。**

### 5. Mock模式入侵性
`recharge/routes.py` 第135行硬编码了 `mock_app_id = "wxb4f6d89904200fd2"`。在 `PAYMENT_MODE=real` 模式下不会执行这段代码（第115行有 `if is_real_mode()` 保护）。**安全。**

### 6. 支付宝状态
支付宝模块虽代码完整，但充值模块目前限制仅支持 `wxpay`（`recharge/routes.py` 第93行）。支付宝要有完整的充值链路需要额外开发。目前支付宝只存在于订单支付的 `routers/payment.py` 中（第424-474行）。
