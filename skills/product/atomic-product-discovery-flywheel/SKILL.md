# 原子产品发现飞轮（Atomic Product Discovery Flywheel）

> 从原子池中系统化发现「一起用才有更大价值」的原子组合→蜂巢讨论三维评分→创建完整产品文件夹的一站SOP

## 触发条件

| 场景 | 触发模式 |
|:-----|:---------|
| 用户说「用蜂巢讨论做产品方案设计」 | ✅ 完整飞轮 |
| 用户说「从原子池找组合」 | ✅ 完整飞轮 |
| 用户说「把XX和YY组合成产品」 | ✅ 可跳过蜂巢讨论，直接走评分+建文件夹 |
| 日常唤醒扫描（铁律十六） | ✅ 自动扫描沉睡原子 |

## 完整五步飞轮

### Step 0: 全景扫描（5分钟）

**目标**：了解原子池现状，不做无依据的排列组合。

```bash
# 扫描 skills/ 目录获取完整原子清单
ls -R $HERMES_HOME/skills/ | grep "SKILL.md" | head -200

# 扫描已有产品文件夹获取atoms使用情况
ls $HERMES_HOME/L5孵化室/产品开发/*/atoms/INDEX.md 2>/dev/null

# 读取最新产品就绪度报告（如有）
cat $HERMES_HOME/L5孵化室/产品开发/_产品就绪度报告.md 2>/dev/null
```

**产出**：原子清单 + 已有产品atoms覆盖情况 + 沉睡原子识别。

---

### Step 1: IDEATE — 并行派3+角色员工独立分析

**强制规则**：至少3个不同角色，各自从专业视角独立输出。

| 角色 | 派单目标 | 输出内容 |
|:-----|:---------|:---------|
| **乘黄（产品经理P8）** | 「从产品价值和用户付费意愿角度，找出至少3个原子组合，三维评分」 | 组合清单+评分+排序 |
| **烛龙（工程师）** | 「核查每个组合中引用的原子是否有真实的运行代码（不是只有文档），给出技术可行性和MVP人日」 | 代码量核查+风险分析+工时估算 |
| **鲲鹏（架构师CTO）** | 「从架构层和竞品角度做冷眼评估，检查这些组合是否和已有产品重叠，给出独创建议」 | 架构评估+竞品对标+候选否决/独创方案 |

**关键判断标准**——「一起用才有更大价值」的定义：
1. 这些原子单独用已有价值，但合在一起才形成完整业务闭环
2. 比如：情报采集+分析+推送SaaS（缺任何一环都不完整）
3. 纯粹把已有产品改个名字算新组合→**不允许**

**三维评分标准**：

| 维度 | 权重 | 评分标准（1-5分） |
|:-----|:----:|:------------------|
| **V1 支付意愿** | 40% | 5=降本增效有清晰ROI / 3=有价值但难量化 / 1=只有个人愿意用 |
| **V2 替代成本** | 30% | 5=替代需招人年薪30万+ / 3=买个SaaS就行 / 1=Excel也能凑合 |
| **V3 交付难度** | 30% | 5=现有原子直接拼 / 3=需少量胶水代码 / 1=需从零开发 |

**产品潜力分 = V1×40% + V2×30% + V3×30%**

---

### Step 2: DEBATE — 观点碰撞

将IDEATE阶段的3份独立产出汇总，识别争议焦点：

| 典型争议 | 仲裁方法 |
|:---------|:---------|
| 「这个组合和已有产品重叠了吗？」 | 蝶检已有产品文件夹的PRODUCT.md定位描述 |
| 「MV的人日估算差距多大？」 | 烛龙代码核查为准（代码事实>任何人估算） |
| 「竞品对标是否真实？」 | 烛龙/鲲鹏的冷眼评估优先于乘黄的产品乐观 |
| 「这个真的有人付钱吗？」 | 先内部验证后对外（不跳过内部验证阶段直接SaaS） |

**鲲鹏否决条款**：当现有代码零行+竞品巨头+架构不可扩展三个条件同时满足时，鲲鹏的否决权优先于乘黄的推荐。

---

### Step 3: CONVERGE — CEO汇总拍板

汇总IDEATE+DEBATE的产出，CEO做出仲裁决策：

**输出格式**：
```
## CEO拍板结果

### 采纳/否决清单

| 组合 | 乘黄分 | 烛龙分 | 鲲鹏评判 | CEO决策 |
|:-----|:------:|:------:|:---------|:--------|
| 组合A | 4.40 | 4.0 | 与䑏疏重叠 | ❌ 否决——已有产品 |
| 组合B | 4.10 | 3.5 | 架构融合困难 | ❌ 否决——集成成本被低估 |
| 组合C (鲲鹏独创) | — | 5.0 | 原创不重叠 | ✅ 采纳——赛博参谋 |

### 采纳产品的三维评分（CEO修正后）
- 组合名：XXX
- V1得分+理由
- V2得分+理由
- V3得分+理由
- 总分：X.XX
```

**拍板标准**：
- 总分≥4.0：自动创建产品文件夹
- 3.0~3.9：创建文件夹，标注"待确认"
- <3.0：仅记录日志，不建文件夹
- 与已有产品≥80%重叠：否决，改为注入方案

---

### Step 4: EXECUTE — 创建产品文件夹（派文鳐执行）

**强制交付物（3个文件，缺一不可）**：

```
$HERMES_HOME/L5孵化室/产品开发/{新产品名}/
├── PRODUCT.md                     # 产品定义
│   ├── 一句话定位
│   ├── 目标用户（含谁在什么场景下为什么付钱）
│   ├── 核心功能（3-5个核心能力）
│   ├── 引用的原子清单（含每个原子在产品中的作用）
│   ├── 定价模式（免费版/专业版/企业版）
│   └── 参考已有产品（避免与现有产品盲目重叠）
├── 价值判断报告.md                # 价值判断
│   ├── 组合发现逻辑（为什么选这些原子）
│   ├── 三维评分明细（V1/V2/V3各自展开+评分理由）
│   ├── 量本利测算（三阶段+盈亏平衡+90天营收估算）
│   └── 风险提示（技术风险/市场风险/竞争风险）
└── atoms/INDEX.md                # 原子索引
    ├── 每个引用原子的名称+链接
    └── 每个原子在产品中的作用描述
```

**验证**：每个文件创建后执行 `ls -la` 和 `wc -l` 确认存在且非空。

---

### Step 5: 记忆沉淀 — 沉淀经验到员工记忆库

1. 将本次产品发现的经验（哪些组合被否决及原因、三维评分决策记录）写入五池
2. 写入MEMORY.md

## 实战案例参考

详见MEMORY.md 2026-05-08记录：赛博参谋（4.30分）和AI灵魂觉醒引擎（4.70分）的完整蜂巢讨论→评分→建文件夹流程。

**关键教训**：
1. 乘黄的产品乐观需要被烛龙和鲲鹏的冷眼对冲
2. 74%的原子组合候选会被否决（因为与已有产品重叠或集成成本被低估）
3. 鲲鹏的否决权在「与已有产品重叠」场景下最高效
4. 3个角色独立分析的总token消耗约200万，但产出的质量远高于CEO一个人拍脑袋
