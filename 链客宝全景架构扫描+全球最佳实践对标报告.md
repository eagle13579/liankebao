## 五、架构质量问题

| 问题 | 类型 | 说明 |
|:-----|:-----|:------|
| 🚨 路由冲突 | ⛓️ 架构 | 网关(:5136)将 `/api/match/` 转发到:8003(数字名片)，导致:8001的matching_engine永远无法被外部访问 |
| 🚨 数据孤岛 | ⛓️ 架构 | 8001用chainke.db，8003用digital_brochure.db，名片数据不通 |
| 🚨 无状态管理 | 🧱 前端 | 无zustand/redux/react-query，只有React context — 复杂状态管理欠缺 |
| 🚨 无API层抽象 | 🧱 前端 | 无axios/fetch封装层，无统一错误处理、无请求拦截器 |
| ⚠️ 匹配引擎单机 | 📊 扩展 | 匹配引擎无集群方案，单点不可扩展 |
| ⚠️ 测试覆盖不均 | 🧪 质量 | 安全测试很全但UI/前端测试仅18个文件 |
| ⚠️ .env硬编码 | 🔐 安全 | GEMINI_API_KEY通过vite的define注入构建产物 |

**路由冲突详情**:
- **位置**: `gateway.py` 第43行 `("/api/match/", "http://localhost:8003", None)`
- **影响**: 所有 `/api/match/*` 请求都被转发到8003数字名片服务
- **后果**: 8001的matching_engine模块(`/api/matching/needs/{id}/products`等)完全不可达
- **解决方案**: 统一网关策略或合并匹配逻辑到单一服务

## 六、P0级别修复建议

### 🥇 P0-1: 三层信任体系 (对标Airbnb)

**当前**: 无评分体系 → 所有用户平等
**目标**:
1. **L1 身份认证层**: 企业工商认证 + 个人实名 + 邮箱验证
2. **L2 交互信誉层**: 匹配后互评(响应速度/合作意愿/信息真实度)
3. **L3 行为信号层**: 活跃度指标(回复率/平均响应时间/完成率)

**匹配分级**:
- 高信誉(>4.5) → Instant Match (自动推荐+直连)
- 中信誉(3.5-4.5) → Assisted Match (需双方确认)
- 低信誉(<3.5) → Manual Match (仅展示不推荐)

**实施步骤**:
1. 添加`enterprise_trust`表(企业信任分)
2. 添加`user_trust`表(用户交互信誉)
3. 实现评分计算引擎(加权平均+时间衰减)
4. 集成到匹配引擎权重计算

### 🥇 P0-2: 开发者生态 (对标Stripe)

**当前**: 无开放API → 无法被集成
**目标**:
1. **L1 嵌入组件**: 名片挂件(1行JS) / 匹配组件
2. **L2 可定制SDK**: React/Vue SDK
3. **L3 全REST API**: OpenAPI规范 + Webhook事件系统
4. 沙箱环境 + 开发者文档门户

**实施步骤**:
1. 生成OpenAPI 3.0规范文档
2. 创建`payment_sdk`正式发布到PyPI
3. 实现Webhook事件系统(订单创建/匹配成功/支付完成)
4. 创建开发者门户(docs.liankebao.top)

### 🥇 P0-3: 设计系统 (对标Linear)

**当前**: Tailwind默认配置 → 无品牌一致性
**目标**:
1. **设计Token**: 颜色/间距/字体/阴影/动画 语义化体系
2. **组件库文档**: Storybook或类似工具
3. **性能基线**: 首屏<1s / 搜索<200ms / 名片加载<100ms
4. 暗色模式

**实施步骤**:
1. 创建`design-tokens.json`定义颜色/间距/字体
2. 集成Storybook记录组件库
3. 实现性能监控(LCP/FID/CLS)
4. 添加暗色模式切换

### 🥇 P0-4: 匹配引擎升级 (对标Airbnb+Einstein)

**当前优势**: 已有TF-IDF+余弦+向量+规则+A/B测试
**当前缺失**:
1. **可解释性层**: "推荐原因: 同行业+同区域+历史成交"
2. **多目标排序**: 预订概率 × 信誉分 × 活跃度 × 距离
3. **动态权重**: 基于历史转化数据自动调权
4. **Bandit探索**: 冷启动用户/新企业的探索机制

**实施步骤**:
1. 添加`explanation`字段到匹配结果
2. 实现多目标排序算法
3. 添加权重学习模块
4. 集成Thompson Sampling冷启动策略

### 🥇 P0-5: 路由冲突修复

**方案**: 统一网关策略
1. `:5136/gateway/api/match/` → `:8001` (主匹配引擎)
2. `:5136/gateway/api/matching/` → `:8003` (数字名片)
**或**: 将两套匹配逻辑合并到`:8001`，废弃`:8003`独立版本

**具体修改**:
```python
# gateway.py 第43行修改
("/api/matching/", "http://localhost:8001", None),  # 主匹配引擎
("/api/match/", "http://localhost:8003", None),    # 数字名片匹配
("/api/brochure/", "http://localhost:8003", None), # 数字名片服务
```
