# 需求评审门禁 — 惊叹点测试

> 来源：一堂业务公式·M10(MVP砍到惊叹点)
> 版本：v2.0（2026-06-06 新增惊叹点门禁）

---

## 原有门禁（保留，不覆盖）

- [ ] 需求背景清晰
- [ ] 目标用户明确
- [ ] 技术方案可行
- [ ] 排期合理

## 新增门禁：惊叹点测试（P0强度）

### 门禁规则

**每个需求在评审时必须回答以下问题：**

```
Q1: 这个功能/产品的"惊叹点"是什么？
    → 用户用了之后会"哇"出来的那个瞬间
    → 不是"好用"、"方便"——是"竟然可以这样"

Q2: 如果只能做一个功能，是这个吗？
    → 如果答案犹豫 → 这个不是惊叹点

Q3: 其他功能砍到P2/P3以后，用户会抱怨少了什么？
    → 如果抱怨的不是这个惊叹点 → 惊叹点还没找到
```

### 门禁流程

```
需求提交 → 标注"惊叹点"
    ↓
评审会
    ├─ 惊叹点清晰且有说服力 → 进入正常评审
    ├─ 惊叹点模糊或无 → 退回重新设计
    └─ 功能列表超过3个P0 → 强制砍到只剩惊叹点相关的功能
```

### 执行铁律

1. 一次迭代只允许1个惊叹点
2. 所有非惊叹点功能自动标记为P2/P3
3. 不通过惊叹点测试的需求不允许进入开发排期

### 示例

```
✅ 正确：
  链客宝名片模块的惊叹点 = "扫码后3秒内看到带实时数据的3D名片，不需要下载APP"
  P0: 3D名片渲染 + 扫码识别
  P2: 名片编辑、历史记录、数据分析

❌ 错误：
  链客宝名片模块的惊叹点 = "好用、方便、功能全面"
  P0: 名片展示+编辑+分享+数据分析+支付+聊天
  → 没有惊叹点，只是在堆功能
```
