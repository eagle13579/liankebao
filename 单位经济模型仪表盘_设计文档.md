## 1. 概述与目标

### 1.1 为什么需要这套仪表盘

链客宝目前处于**从 0 到 1 的增长阶段**，核心业务是连接企业家供需双方（企业家发需求 / 服务商接单）。当前存在以下问题：

- **没有系统化的 LTV / CAC 跟踪**：无法判断获客投入是否合理
- **定价缺乏量化基准**：不清楚一个客户应该收多少钱才能覆盖获客成本
- **流失缺乏监控**：不知道月度留存是否恶化
- **无法做增长预算决策**：加大投放前没有数据支撑

### 1.2 这套仪表盘要解决的核心问题

| 问题 | 回答 |
|------|------|
| 每个客户平均能带来多少毛利？ | LTV |
| 获取一个客户要花多少钱？ | CAC |
| 获客投入划算吗？ | LTV / CAC 比值 |
| 多久能收回获客成本？ | CAC 回收周期 |
| 产品本身盈利吗？ | 毛利率 |
| 客户留得住吗？ | 月流失率 |

### 1.3 适用边界

- **当前版本**只覆盖 C 端企业家用户（需求方）的单位经济模型
- **后续版本**扩展至 B 端服务商的单位经济模型
- 数据颗粒度为**月度**，后续可细化到周度

## 2. LTV 计算模型

### 2.1 标准公式

```
LTV = ARPU × 毛利率 × (1 / 月流失率)
```

### 2.2 各变量定义

#### ARPU（Average Revenue Per User）— 每用户平均月收入

| 项目 | 内容 |
|------|------|
| **口径** | 每月从**付费用户**处获得的总收入 ÷ 当月付费用户数 |
| **包含哪些收入** | 会员订阅费、发布需求付费、置顶/加急等增值服务费、成交抽佣 |
| **不包含哪些收入** | 广告收入（如有）、B 端服务商入驻费（如有）— 这些在另一个业务线 |
| **数据源** | 订单表（payments / transactions），按 user_id 聚合 |
| **SQL 示例口径** | `SUM(payment_amount) / COUNT(DISTINCT user_id)` where `payment_date` 在当月 |

> **链客宝当前情况**：如果目前以免费模式为主、付费渗透率较低，ARPU 会偏低。建议先追踪**付费用户 ARPU** 和**整体 ARPU** 两条线。

#### 毛利率（Gross Margin）

| 项目 | 内容 |
|------|------|
| **口径** | (收入 − 直接成本) ÷ 收入 × 100% |
| **包含哪些成本** | 服务器/云成本、支付通道手续费、AI 调用成本（GPT token 等）、客服人力分摊、内容审核人力分摊 |
| **不包含哪些成本** | 市场费用、销售薪资、研发薪资（这些是运营费用，不是 COGS） |
| **SaaS 行业基准** | 通常 > 70%，优秀 > 80%，链客宝建议目标 > 60%（含 AI 成本） |
| **数据源** | 云账单（阿里云/AWS）、支付渠道对账单、AI API 用量报表、财务系统 |
| **计算周期** | 月度 |

> **特别说明**：链客宝是 AI 驱动的平台，GPT/Claude 等 API 调用是重要的直接成本项，务必纳入毛利率计算。

#### 月流失率（Monthly Churn Rate）

| 项目 | 内容 |
|------|------|
| **口径** | 当月流失的付费用户数 ÷ 月初付费用户数 × 100% |
| **"流失"定义** | 连续 30 天（或 1 个完整自然月）无任何付费行为 = 流失 |
| **新用户处理** | 本月新增的付费用户不计入月初基数，但计入当月流失计算的分母需做调整（见下方详细说明） |
| **SaaS 行业基准** | 月流失率 3–5% 为健康，> 7% 需警惕 |
| **数据源** | 用户付费行为表（user_payments / subscription_status） |
| **精确公式** | `Churn = Lost_Customers_in_Month / Customers_at_Start_of_Month` |

> **链客宝适应**：如果链客宝目前不是订阅制而是按次付费，流失定义需调整：**"连续 90 天无登录 + 无付费行为"** 视为流失。初始阶段建议用 90 天窗口。

### 2.3 完整 LTV 示例计算

假设链客宝某月数据如下：

| 变量 | 数值 | 说明 |
|------|------|------|
| 月度总收入 | ¥50,000 | 所有付费用户当月付款总额 |
| 付费用户数 | 200 | 当月至少有 1 笔付款的用户 |
| **ARPU** | **¥250** | 50,000 ÷ 200 |
| 月直接成本 | ¥15,000 | 云服务 ¥5,000 + AI API ¥6,000 + 支付手续费 ¥1,000 + 客服分摊 ¥3,000 |
| **毛利率** | **70%** | (50,000 − 15,000) ÷ 50,000 |
| 月初付费用户 | 200 | |
| 当月流失用户 | 12 | |
| **月流失率** | **6%** | 12 ÷ 200 |
| **LTV** | **¥2,917** | 250 × 0.70 × (1 ÷ 0.06) |

**解读**：每个付费用户在整个生命周期内平均贡献 ¥2,917 的毛利。如果获客成本（CAC）低于 ¥972，则 LTV/CAC > 3，健康。

### 2.4 LTV 计算注意事项

1. **新用户与老用户分开跟踪**：新用户（首月）的流失率天然高，混合计算会压低 LTV。建议分群：
   - 新用户 LTV（首月组）
   - 成熟用户 LTV（第 3 个月及以上）
2. **按 cohort（同期群）追踪**：每月进入的一批用户，追溯到 12 个月后看累计 LTV，比单月截面数据更准确
3. **折扣率**：如果做 LTV 的 DCF（折现）版本，按月度折现率 1%（年化约 12.68%）将未来现金流折现

## 3. CAC 计算模型

### 3.1 标准公式

```
CAC = 营销费用 / 新增客户数
```

### 3.2 两种 CAC 口径

#### 付费 CAC（Paid CAC）

| 项目 | 内容 |
|------|------|
| **分子** | 付费渠道支出（信息流广告、SEM/SEO、KOL 投放、付费内容分发） |
| **分母** | 源自付费渠道的新增付费用户数（首次完成付费） |
| **用途** | 评估付费渠道的获客效率，做渠道 ROI 对比 |
| **数据源** | 广告平台后台 + UTM 参数追踪 + 用户归因表 |

#### 混合 CAC（Blended CAC）

| 项目 | 内容 |
|------|------|
| **分子** | 全部营销费用（付费渠道 + 内容营销团队薪资 + 社群运营 + 线下活动 + 品牌公关 + SaaS 工具订阅） |
| **分母** | 全部新增付费用户数（无论来源） |
| **用途** | 评估整体获客效率，用于 LTV/CAC 比值计算 |
| **数据源** | 财务系统（按营销费用科目汇总）+ 用户注册/付费表 |

> **链客宝当前建议**：初期先跟踪**混合 CAC**，因为链客宝目前的获客以内容/社群/口碑为主，付费渠道占比低。等付费投放规模化后再分开拆分付费 CAC。

### 3.3 营销费用拆解

| 费用类别 | 是否计入 CAC | 说明 |
|----------|-------------|------|
| 信息流广告（抖音/朋友圈） | ✅ | 可追踪到具体用户 |
| SEO / ASO | ✅ | 含工具费 + 执行人力按比例分摊 |
| KOL / KOC 合作 | ✅ | 按条/按 ROI 结算的部分 |
| 内容团队薪资 | ✅（部分） | 按内容获客贡献比例分摊（初期可按 50% 估算） |
| 社群运营薪资 | ✅（部分） | 社群拉新相关的部分 |
| 线下展会/活动 | ✅ | 仅计入拉新导向的活动费用 |
| 品牌公关 | ⚠️ 暂不计入 | 品牌类非直接获客，可单独追踪 |
| 销售团队底薪 | ⚠️ 暂不计入 | B 端销售单独核算，初期先做 C 端 CAC |
| SaaS 工具（飞书/企微等） | ✅（小额） | 按营销团队人头分摊 |

### 3.4 CAC 示例计算

| 项目 | 金额 |
|------|------|
| 信息流广告投放 | ¥20,000 |
| 内容营销人力（50% 分摊） | ¥15,000 |
| KOL 合作费用 | ¥8,000 |
| 社群运营工具 | ¥2,000 |
| **总营销费用** | **¥45,000** |
| 月新增付费用户 | 60 |
| **混合 CAC** | **¥750** |

**对比 LTV（上例 ¥2,917）**：LTV/CAC = 2,917 ÷ 750 ≈ 3.89 ✅ 健康

### 3.5 CAC 回收周期

```
CAC 回收周期（月）= CAC / (ARPU × 毛利率)
```

示例：CAC = ¥750，ARPU = ¥250，毛利率 = 70%

```
回收周期 = 750 / (250 × 0.70) = 750 / 175 ≈ 4.3 个月
```

**解读**：4.3 个月可以收回获客成本，远低于 12 个月的警戒线，非常健康。

### 4.4 看板数据表结构

```sql
CREATE TABLE unit_economics_metrics (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    report_month DATE NOT NULL,             -- 数据月份，如 '2026-06-01'
    arpu DECIMAL(12,2),                      -- 付费用户 ARPU
    arpu_all DECIMAL(12,2),                  -- 整体用户 ARPU
    gross_margin DECIMAL(5,4),               -- 毛利率，如 0.7000
    monthly_churn_rate DECIMAL(5,4),         -- 月流失率，如 0.0600
    ltv DECIMAL(12,2),                       -- LTV
    marketing_spend DECIMAL(12,2),           -- 营销费用总额
    new_paid_users INT,                      -- 新增付费用户数
    blended_cac DECIMAL(12,2),               -- 混合 CAC
    paid_cac DECIMAL(12,2),                  -- 付费 CAC（可为 NULL）
    ltv_cac_ratio DECIMAL(8,2),              -- LTV / CAC 比值
    cac_payback_months DECIMAL(6,2),         -- CAC 回收周期（月）
    active_paid_users INT,                   -- 当前活跃付费用户数
    total_revenue DECIMAL(12,2),             -- 月度总收入
    direct_cost DECIMAL(12,2),               -- 月度直接成本
    churned_users INT,                       -- 当月流失用户数
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    updated_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP ON UPDATE CURRENT_TIMESTAMP,
    UNIQUE KEY uk_month (report_month)
);
```

## 5. 健康阈值体系

### 5.1 三档阈值总览

| 指标 | 🟢 健康（绿色） | 🟡 关注（黄色） | 🔴 危险（红色） | 参考行业标准 |
|------|:---------------:|:---------------:|:---------------:|:------------:|
| **LTV / CAC** | > 3.0 | 2.0 – 3.0 | < 2.0 | SaaS 基准 > 3 |
| **CAC 回收周期** | < 6 个月 | 6 – 12 个月 | > 12 个月 | SaaS 基准 < 12 个月 |
| **毛利率** | > 60% | 40% – 60% | < 40% | SaaS 基准 > 50% |
| **月流失率** | < 5% | 5% – 8% | > 8% | SaaS 基准 < 5% |
| **ARPU（月）** | > ¥200 | ¥100 – ¥200 | < ¥100 | 取决于客群定位 |
| **CAC** | < ¥1,000 | ¥1,000 – ¥2,000 | > ¥2,000 | 取决于获客渠道成本 |

### 5.2 阈值设定依据

**LTV/CAC > 3（绿色）**：
- 一堂业务标准：LTV/CAC > 3 健康
- 逻辑：每赚 3 元只花 1 元获客，留出足够空间覆盖运营费用和研发费用
- 链客宝初始设定：> 3 绿，2–3 黄，< 2 红

**CAC 回收周期 < 6 个月（绿色）**：
- SaaS 行业标准：< 12 个月即及格
- 链客宝初始阶段现金流敏感，收紧到 6 个月为健康，12 个月为警戒线

**毛利率 > 60%（绿色）**：
- 考虑到链客宝有 AI API 成本，比纯软件 SaaS 毛利率偏低
- 设定 60% 为健康线（参考：一堂 SaaS > 50%，链客宝微调）

**月流失率 < 5%（绿色）**：
- SaaS 标准 5–7% 为一般水平
- 链客宝早期用户基数小，流失率波动大，暂定 5% 为健康线

### 5.3 综合健康评分

```
综合健康评分 = 按指标的权重加权平均

权重建议：
  - LTV/CAC: 30%（最重要的北极星指标）
  - CAC 回收周期: 20%
  - 毛利率: 20%
  - 月流失率: 20%
  - ARPU: 10%

每项指标得分：绿色=100, 黄色=60, 红色=20

综合健康评分 = Σ(指标权重 × 指标得分)
```

| 综合评分 | 状态 |
|---------|------|
| ≥ 85 | 🟢 健康 |
| 60 – 84 | 🟡 需关注 |
| < 60 | 🔴 危险 |

### 5.4 阈值调整机制

- **每季度**复盘阈值设定
- 链客宝进入增长期（月活 > 10,000 / 月收入 > ¥50 万）后，阈值按行业标准收紧
- 链客宝早期阶段（月收入 < ¥10 万），红色阈值可以先放松 30%，避免频繁误报

## 6. 数据源说明

### 6.1 当前已有数据源

| 数据源 | 包含内容 | 状态 | 能否支撑仪表盘 |
|--------|---------|------|:-------------:|
| **订单/支付表** | 用户付款记录、金额、时间、支付方式 | ✅ 已有 | ✅ 核心 — ARPU、收入 |
| **用户表** | 注册信息、用户类型（企业家/服务商） | ✅ 已有 | ✅ 基础 — 用户分组 |
| **行为日志** | 登录、浏览、发布需求、接单等事件 | ✅ 已有 | ⚠️ 用于定义"活跃" |

## 7. 预警机制

### 7.1 预警触发规则

每天 08:00 检查最新月份的 6 个核心指标，按以下规则触发预警：

| 指标 | 黄色预警（Notify） | 红色预警（Alert） |
|------|:-----------------:|:-----------------:|
| LTV / CAC | 2.0 – 3.0 | < 2.0 |
| CAC 回收周期 | 6 – 12 个月 | > 12 个月 |
| 毛利率 | 40% – 60% | < 40% |
| 月流失率 | 5% – 8% | > 8% |
| ARPU | ¥100 – ¥200 | < ¥100 |
| CAC | ¥1,000 – ¥2,000 | > ¥2,000 |

### 7.2 预警触发动作

#### 黄色预警动作（需关注）

| 动作 | 责任人 | 时限 |
|------|--------|:----:|
| 推送飞书/企业微信消息到「增长数据群」 | 自动化 | 即时 |
| 消息模板：「⚠️ 链客宝预警：[指标名] 进入黄色区域，当前值 [X]，上月值 [Y]，请关注」 | — | — |
| 自动在仪表盘页面标注该指标为黄色 | 系统 | 即时 |
| 记录预警日志到 `alert_logs` 表 | 自动化 | 即时 |
| PM 需要在 3 个工作日内回复原因分析 | PM | 3 天 |
| 触发后连续 3 个月指标未改善 → 升级为红色预警 | 自动 | 3 个月 |

#### 红色预警动作（需立即行动）

| 动作 | 责任人 | 时限 |
|------|--------|:----:|
| 推送飞书/企业微信消息到「管理群」（含 CEO/CTO/COO） | 自动化 | 即时 |
| 消息模板：「🚨 链客宝紧急预警：[指标名] 进入红色区域，当前值 [X]，历史趋势 [Y→Z]，请立即处理」 | — | — |
| 自动发送邮件到核心管理层邮箱 | 自动化 | 即时 |
| 在仪表盘页面弹窗提示「严重预警」| 系统 | 即时 |
| 触发**获客预算冻结**：暂停所有付费渠道的新增投放，保留有机增长渠道 | 运营负责人 | 24 小时内 |
| 启动**紧急复盘会议**：CEO / CTO / COO / PM / 运营 共同参与 | CEO | 48 小时内 |
| 形成**改进计划文档**，明确改进措施、责任人、预期恢复时间 | PM | 72 小时内 |

### 7.3 预警详细场景指南

#### 场景 A：LTV/CAC 进入红色（< 2.0）

**可能原因**：
1. CAC 急剧上升（投放成本变高、新渠道效率低）
2. LTV 下降（ARPU 下降 / 流失率上升 / 毛利率下降）
3. 两者同时恶化

**建议行动**：
1. 冻结所有付费投放（直接降低 CAC）
2. 分析流失原因 — 产品功能缺失？价格不合理？竞品出现？
3. 提高 ARPU — 推出更高客单价的付费套餐
4. 降低成本结构 — 优化 AI 调用（缓存、模型蒸馏）

#### 场景 B：月流失率进入红色（> 8%）

**可能原因**：
1. 产品体验不佳（注册后不知道该做什么）
2. 需求匹配率低（用户发布需求后没有合适的服务商响应）
3. 竞品推出更有竞争力的产品

**建议行动**：
1. 对流失用户做回访（每流失 100 个用户至少电话/微信回访 10 个）
2. 分析流失用户的行为漏斗 — 他们在第几步流失？
3. 强化新用户 onboarding 流程
4. 提升需求响应速度和匹配准确率

#### 场景 C：毛利率进入红色（< 40%）

**可能原因**：
1. AI API 调用成本失控
2. 服务器资源未被合理利用
3. 定价过低

**建议行动**：
1. 审查 AI 调用策略（是否过多无效调用？能否用更便宜的模型？）
2. 考虑提价或推出新的付费套餐
3. 优化云资源（缩扩容策略、竞价实例）

### 7.4 预警日志表结构

```sql
CREATE TABLE alert_logs (
    id BIGINT AUTO_INCREMENT PRIMARY KEY,
    alert_type VARCHAR(20) NOT NULL,         -- 'YELLOW' 或 'RED'
    metric_name VARCHAR(50) NOT NULL,         -- 指标名称（ltv_cac_ratio / churn_rate 等）
    metric_value DECIMAL(12,4) NOT NULL,      -- 触发时的指标值
    threshold_value DECIMAL(12,4) NOT NULL,   -- 触发的阈值边界
    report_month DATE NOT NULL,               -- 数据月份
    created_at TIMESTAMP DEFAULT CURRENT_TIMESTAMP,
    resolved_at TIMESTAMP NULL,               -- 解除时间
    resolution_note TEXT,                     -- 解除说明
    INDEX idx_month (report_month),
    INDEX idx_resolved (resolved_at)
);
```

### 7.5 预警抑制规则

- **同指标同月份**不重复触发预警（避免每月 1 日 ETL 跑完后反复推送）
- 指标从黄色改善到绿色 → 发送一条恢复通知：「✅ 链客宝通知：[指标名] 已恢复至绿色区域」
- 指标从红色改善到黄色 → 发送一条降级通知：「⚠️ 链客宝通知：[指标名] 从红色降至黄色，仍需关注」
- 连续 6 个月绿色状态的指标 → 自动调高该指标的绿色阈值（收紧标准）
